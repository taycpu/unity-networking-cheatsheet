◈ PROTOCOL BATTLE CHART

Protocol Transport Latency Reliability Best For (Unity) Avoid When
TCP Connection-based ~10–50ms ✓ Guaranteed order Chat, login, inventory, matchmaking Player movement, FPS
UDP Connectionless ~1–5ms ✗ Fire-and-forget Position sync, projectiles, voice Financial txns, auth
WebSocket TCP + HTTP upgrade ~15–40ms ✓ Ordered (TCP) Web/mobile clients, lobby, events Sub-10ms game state
QUIC/HTTP3 UDP-based streams ~5–20ms ✓ Multi-stream Modern WebTransport, future Broad platform support needed

⚠️ BOSS MECHANIC: Head-of-Line Blocking

TCP guarantees order. If packet 4 is lost, packets 5–99 sit and wait. For a 60Hz game loop, this is catastrophic.

UDP skips dropped packets. Your game code decides: "I don't need that old position update, I'll use the new one."

◈ WHEN TO USE WEBSOCKETS IN UNITY

✓ USE WebSockets when:

  • • Client is a WebGL build (no raw UDP)
  • • Events are infrequent (lobby state, scores)
  • • You're behind a corporate firewall/proxy
  • • Building a companion web/mobile app
  • • You need server → client push notifications

✗ SKIP WebSockets when:

  • • Sending 60 packets/sec per player
  • • You need sub-10ms jitter tolerance
  • • FPS / racing / fighting game physics
  • • Running on dedicated server (use UDP + NGO)

◈ WHY PROTOBUFS DESTROY JSON IN GAMES

📄 JSON (what you know)

// JSON PlayerState: ~220 bytes { "playerId": "abc-123-def", "position": { "x": 123.456, "y": 0.0, "z": -88.901 }, "timestamp": 1714393200000, "health": 100 }

⚡ Protobuf (what you want)

// Same data: ~28 bytes (8x smaller!) // Binary: field tags + varint encoding 0A 0B 61 62 63 ... // player_id 15 79 E9 F6 42 // x = 123.456 1D 00 00 00 00 // y = 0.0 25 1B 2F B1 C2 // z = -88.901 28 C0 A8 A6 B2 06 // timestamp 30 64 // health = 100
Metric JSON Protobuf Advantage
Payload size 100% ~12–30% 3–8x smaller
Parse speed Slow (text) Very fast (binary) ~5–10x faster
Schema Optional (loose) Required (.proto) Enforced contracts
Readability Human-readable Binary (not readable) JSON wins
Versioning Manual Field numbers → safe evolve Backward-compat

🏆 LEGENDARY TIP: Field Number Rule

Field numbers in Protobuf are permanent wire identifiers. Once you ship with field 3 = "position", you can NEVER change that mapping. You can add new fields (field 8, 9…), but never reorder or delete without `reserved`. This is how Protobuf stays backward-compatible across server versions.

◈ gRPC: THE MULTIPLAYER ARCHITECT'S SWISS ARMY KNIFE

gRPC = Protobuf (serialization) + HTTP/2 (transport) + generated stubs (no manual parsing).

It gives you 4 call types:

Unary One request → one response. Like REST.
Server Stream One request → stream of responses. Game snapshots!
Client Stream Stream of requests → one response. Upload replay data.
Bidi Stream Full duplex streams both ways. Real-time game channel!
View: .proto service definition
// matchmaking.proto service MatchmakingService { // Unary: find a match rpc FindMatch (MatchRequest) returns (MatchResult); // Server stream: receive game snapshots rpc SubscribeToGame (GameSubscribeRequest) returns (stream GameSnapshot); // Bidi stream: full game channel rpc GameChannel (stream ClientInput) returns (stream GameSnapshot); }
View: Unity C# gRPC client
var channel = GrpcChannel.ForAddress("https://server:9090"); var client = new MatchmakingService.MatchmakingServiceClient(channel); // Unary call var match = await client.FindMatchAsync(new MatchRequest { PlayerId = localId, Rank = playerRank }); // Server-streaming: receive game snapshots using var stream = client.SubscribeToGame( new GameSubscribeRequest { GameId = match.GameId }); await foreach (var snapshot in stream.ResponseStream.ReadAllAsync()) { ApplyGameSnapshot(snapshot); // update all players }

◈ MONOLITH vs MICROSERVICES

🏚️ Monolith (how most game backends start)

GameServer.exe { Matchmaking + Inventory + Leaderboard + Auth + Chat + Anti-cheat + Analytics } // One crash kills everything. // Scale matchmaking? Scale it ALL.

  • • Fault isolation (one crash ≠ game down)

◈ HOW THEY SCALE: THE GOLDEN RULES

📈 Horizontal Scaling

Run N copies of a service behind a load balancer. Works when services are stateless.

Auth, Matchmaking, REST APIs → stateless → scale easily

📦 Stateful Services

Game sessions are stateful. Each game lives on ONE server. Use consistent hashing to route players to their session.

Game Session Service → sticky routing by game_id

🗄️ Database Per Service

Each microservice owns its data. No shared DB. Communicate via APIs and events only.

Inventory: PostgreSQL. Leaderboard: Redis. Chat: Cassandra.

🔒 Service Mesh

In Kubernetes, services talk via a mesh (Istio/Linkerd). mTLS between services, circuit breaking, retries.

You don't build this — you configure it.

THE ORACLE — PLAIN ENGLISH ANALOGIES

⚡ TCP vs UDP — The Pizza Delivery

TCP is a pizza delivery with signature required. If you're not home (packet lost), the driver waits, and ALL other deliveries on that truck WAIT too. Perfect for your wallet (bank transactions). Terrible for a live sports broadcast.

UDP is your TV signal. If one frame drops, your TV doesn't freeze waiting for it — it just shows the next frame. A slightly glitchy game > a frozen one.

📦 Protobuf vs JSON — Spoken Language vs Morse Code

JSON is texting in full English sentences: "Hey, my name is player number 42 and I'm at position X equals 123.456..." — readable, but wordy.

Protobuf is Morse code agreed in advance: both sides have a codebook (the .proto file) so "· — · ·" means "field 3 is a float" with no extra context bytes needed. Tiny and blazing fast to decode.

🔄 REST vs Pub/Sub vs gRPC — Types of Communication

REST = making a phone call. You dial, they answer, conversation over. Great for one-off questions.

Pub/Sub = a radio broadcast. The DJ (publisher) talks; anyone tuned in (subscriber) hears it. The DJ doesn't know who's listening. Perfect for "player died" events.

gRPC = a walkie-talkie with a shared channel. Both sides can talk at any time (bidi stream), it's fast (binary), and you have a "channel protocol" so nobody sends garbage.

🏗️ Microservices — A Restaurant Kitchen

A monolith is one chef doing everything: takes orders, cooks, plates, washes dishes, manages the register. If they get sick, the restaurant closes.

Microservices are a proper kitchen brigade. The grill chef (Game Session Service) doesn't care what the pastry chef (Leaderboard Service) is doing. If the dishwasher calls in sick, you still serve food. You hire 3 more grill chefs on Friday night (horizontal scaling) without touching any other station.

The head waiter is your API Gateway — single point of contact for all guests (clients).

⭐ Authoritative Server — The Referee Analogy

In a naive setup, players shout "I shot him! I moved here!" and the server trusts them. That's how cheaters win.

With an authoritative server, players only send inputs ("I pressed W and Space at tick 1042"). The server runs the simulation and shouts back "Here's where you are." Players can predict their own movement (client-side prediction), but the server's word is final. Cheating becomes pointless because your client never controls the ground truth.
