⚠️ BOSS MECHANIC: Head-of-Line Blocking

TCP guarantees order. If packet 4 is lost, packets 5–99 sit and wait. For a 60Hz game loop, this is catastrophic.

UDP skips dropped packets. Your game code decides: "I don't need that old position update — I'll use the new one."

View: Unity C# UDP Send Example 
// Unity: raw UDP for position sync
using System.Net.Sockets;
using Google.Protobuf;

public class UdpPositionSender : MonoBehaviour {
    private UdpClient udp = new UdpClient();
    private IPEndPoint server = new IPEndPoint(IPAddress.Parse("10.0.0.1"), 7777);

    void FixedUpdate() {
        // Build protobuf message (see Module 2)
        var msg = new PlayerState {
            PlayerId  = localPlayerId,
            Position  = new Vec3 { X = transform.position.x,
                                   Y = transform.position.y,
                                   Z = transform.position.z },
            Timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
        };
        byte[] bytes = msg.ToByteArray(); // ~28 bytes vs ~200 bytes JSON
        udp.Send(bytes, bytes.Length, server);
    }
}

◈ WHEN TO USE WEBSOCKETS IN UNITY

✓ USE WebSockets when:

  • • Client is a WebGL build (no raw UDP available)
  • • Events are infrequent (lobby state, scores)
  • • You're behind a corporate firewall / proxy
  • • Building a companion web or mobile app
  • • You need server → client push notifications

✗ SKIP WebSockets when:

  • • Sending 60 packets/sec per player
  • • You need sub-10ms jitter tolerance
  • • FPS / racing / fighting game physics
  • • Running on dedicated server (use UDP + NGO)

◈ WHY PROTOBUFS DESTROY JSON IN GAMES

📄 JSON — what you know (~200 bytes)

{
  "playerId": "abc-123-def",
  "position": { "x": 123.456, "y": 0.0, "z": -88.901 },
  "timestamp": 1714393200000,
  "health": 100
}

⚡ Protobuf — what you want (~28 bytes)

// Binary wire format — 8x smaller!
// field tag + varint / fixed32 encoding
0A 0B 61 62 63 ...  // player_id (string)
15 79 E9 F6 42      // x = 123.456 (fixed32)
1D 00 00 00 00      // y = 0.0
25 1B 2F B1 C2      // z = -88.901
28 C0 A8 A6 B2 06   // timestamp (varint)
30 64               // health = 100

Parse speedSlow (text scan)Very fast (binary)~5–10x faster

◈ YOUR FIRST .PROTO FILE

// game.proto — shared between Unity client and Go/C# server
syntax = "proto3";
package game;
option csharp_namespace = "Game.Proto";

// Field numbers are PERMANENT. Never reuse a deleted number.
message Vec3 {
    float x = 1;
    float y = 2;
    float z = 3;
}

message PlayerState {
    string player_id = 1;
    Vec3   position  = 2;
    Vec3   rotation  = 3;
    float  speed     = 4;
    int32  health    = 5;
    int64  timestamp = 6;
    // Field 7 was "ammo" — deleted. NEVER reuse 7!
    reserved 7; reserved "ammo";
}

message GameSnapshot {
    repeated PlayerState players = 1; // all players in frame
    int64 server_tick = 2;
}

View: Unity C# serialize / deserialize 
using Game.Proto;
using Google.Protobuf;

public class StateSerializer {

    // C# object → binary bytes (called every FixedUpdate)
    public static byte[] Serialize(Transform t, string id) {
        var state = new PlayerState {
            PlayerId = id,
            Position = new Vec3 {
                X = t.position.x,
                Y = t.position.y,
                Z = t.position.z
            },
            Rotation = new Vec3 {
                X = t.eulerAngles.x,
                Y = t.eulerAngles.y,
                Z = t.eulerAngles.z
            },
            Timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
        };
        return state.ToByteArray(); // ~28 bytes 🔥
    }

    // bytes → C# object (called on receive)
    public static PlayerState Deserialize(byte[] data) {
        return PlayerState.Parser.ParseFrom(data);
    }
}

🏆 LEGENDARY TIP: Field Number Rule

Field numbers are permanent wire identifiers. Once you ship field 3 = "position", that mapping is forever. You can add new fields (8, 9…), but never reorder or delete without reserved. This is how Protobuf stays backward-compatible when clients and servers are on different versions.

◈ gRPC — THE MULTIPLAYER ARCHITECT'S SWISS ARMY KNIFE

gRPC = Protobuf + HTTP/2 + generated stubs. Four call types:

UnaryOne request → one response. Like REST.
Server StreamOne request → stream of responses. Perfect for game snapshots.
Client StreamStream of requests → one response. Upload replay data.
Bidi StreamFull duplex streams both ways — real-time game channel.
View: .proto service definition 
// matchmaking.proto
service MatchmakingService {
    // Unary: find a match
    rpc FindMatch (MatchRequest)
        returns (MatchResult);

    // Server stream: receive game snapshots
    rpc SubscribeToGame (GameSubscribeRequest)
        returns (stream GameSnapshot);

    // Bidi stream: full real-time game channel
    rpc GameChannel (stream ClientInput)
        returns (stream GameSnapshot);
}
View: Unity C# gRPC client 
var channel = GrpcChannel.ForAddress("https://server:9090");
var client  = new MatchmakingService.MatchmakingServiceClient(channel);

// Unary call
var match = await client.FindMatchAsync(new MatchRequest {
    PlayerId = localId, Rank = playerRank
});

// Server-streaming: receive game snapshots
using var stream = client.SubscribeToGame(
    new GameSubscribeRequest { GameId = match.GameId });

await foreach (var snapshot in stream.ResponseStream.ReadAllAsync()) {
    ApplyGameSnapshot(snapshot); // update all remote players
}

◈ SCALING GOLDEN RULES

📈 Horizontal Scaling

Run N copies behind a load balancer. Works only when services are stateless. Auth, Matchmaking, REST APIs → stateless → scale trivially.

📦 Stateful Services

Game sessions hold state. Each game lives on ONE instance. Use consistent hashing to route all players for a game_id to the same pod.

🗄️ Database Per Service

Each service owns its data — no shared DB. Share data only via APIs and events. Inventory: PostgreSQL. Leaderboard: Redis. Chat: Cassandra.

🔒 Service Mesh

In Kubernetes, an Istio/Linkerd mesh handles mTLS between services, circuit breaking, and retries automatically. You configure it, not build it.

THE ORACLE — PLAIN ENGLISH ANALOGIES

⚡ TCP vs UDP — The Pizza Delivery

TCP is delivery with signature required. If you're not home (packet lost), the driver waits — and every other delivery on that truck waits too. Perfect for bank transactions. Catastrophic for a live game at 60 FPS.

UDP is your TV signal. A dropped frame? The TV shows the next frame anyway. A slightly glitchy game is better than a frozen one.

📦 Protobuf vs JSON — Spoken Language vs Morse Code

JSON is texting in full sentences: "Hey, my name is Player 42 and I'm at position X equals 123.456…" — readable, but verbose on every single message.

Protobuf is Morse code with a shared codebook (the .proto file). Both sides already know "·—·" means "field 3 is a float". No extra context bytes. Tiny and blazing fast to decode.

🔄 REST vs Pub/Sub vs gRPC — Types of Communication

REST = making a phone call. You dial, they answer, call ends. One-off questions.

Pub/Sub = a radio broadcast. The DJ (publisher) talks; anyone tuned in (subscriber) hears. The DJ doesn't know who's listening — perfect for "player killed" events that 3 different services care about.

gRPC = a walkie-talkie on a shared secure channel. Both sides can talk at any time (bidi stream), everything is binary and typed — no garbage in, no garbage out.

🏗️ Microservices — A Restaurant Kitchen

A monolith is one chef doing everything: orders, cooking, plating, dishes, register. That chef gets sick — restaurant closes.

Microservices are a proper kitchen brigade. The grill chef (Game Session Service) doesn't care what the pastry chef (Leaderboard Service) is doing. Dishwasher calls in sick? You still serve food. On Friday night you hire 3 more grill chefs (horizontal scaling) without touching any other station.

The head waiter is your API Gateway — single point of contact for all guests (clients).

⭐ Authoritative Server — The Referee

Naively, players shout "I shot him! I'm here!" and the server trusts them. That's how cheaters win.

With an authoritative server, players only send inputs: "I pressed W and Space at tick 1042." The server runs the physics simulation and broadcasts back the definitive world state. Client-side prediction keeps it feeling responsive, but the server's word is final. Cheating is pointless because your client never controls the ground truth.
